## 1. Spring Security
* 인증 -> 로그인
    * 유저가 누구인지 확인하는 절차
* 인가 -> 리소스 접근 제어
    * 유저에 대한 권한을 허락하는 것
* 이러한 인증 / 인가를 Spring 환경에서는 Spring Security가 담당한다.

## 2. 인증 / 인가
* 인증 : Spring Security가 신분증을 확인 -> 출입증 발급
* 인가 : Spring Security가 출입증을 확인
* 서블릿 필터 (Servlet Filter) : 요청/응답을 가로채서 전처리/후처리 하는 장치
* Spring Security의 원리
    * 보안을 담당하는 서블릿 필터를 끼워넣는다.
    * 요청과 응답을 가로채서 전처리 및 후처리를 하는 장치
    * (기본적으로 스프링 부트에서도 사용하기도 한다.)
* Spring Security에서는 SecurityFilterChain이라는 이름으로 불린다.

사용자가 로그인 폼 제출 -> 필터가 직접 확인하는 것이 아닌 매니저에게 인증을 위임 -> 실제 DB 조회 -> 결과 반환 구조로 이루어진다.

DTO를 기본 클래스 타입으로 설계하는 것이 아닌 Record 타입으로 설계하면 여러가지 이점을 얻는다.

## 3. Record 타입
* Record
    * DTO를 만드는데 특화되어 있는 class type
        * DTO : Data Transfer Object, 계층간 데이터 교환을 위해 사용하는 객체
        * 생성자, getters, equals, hashCode, toString 메서드 자동 제공
        * 불변 객체로 abstract로 선언 불가, 암시적으로 final로 선언
        * 한번 값이 정해지면 setter를 통해 값을 변경할 수 없다 -> 상속 불가
        * 내부 각 필드 (컴포넌트)는 private final로 정의된다.

    * 장점
        * 간결성 : 가독성이 좋다는 가장 큰 장점이 존재 이해가 쉽고 간결하다 생성자, getters 등 기본 제공되는 메소드들 덕분에 DTO 특성의 클래스를 개발할때 매번 개발자가 직접 구현해야 했던 불필요한 작업들을 줄여준다.
        * 불변성 : 모든 컴포넌트가 final로 선언되며 setter가 금지되어 있기에 불변성을 보장
        * 패턴 매칭 통합 : java의 패턴 매칭과 함께 사용될 수 있어, 데이터 추출 및 패턴 일치 검사 등 모두 사용 가능
        * "데이터 전달"이란 목적을 명시 : "해당 클래스는 데이터 전달 용도임"을 명시적으로 나타낼 수 있으므로 코드에 대한 이해를 돕는다
